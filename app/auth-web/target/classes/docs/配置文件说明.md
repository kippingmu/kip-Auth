# Spring Cloud 配置文件说明
`bootstrap.yml`（或 `bootstrap.properties`）是 Spring Cloud 项目中一个**特殊且优先级更高**的配置文件，主要用于**应用启动初期、Spring 应用上下文（ApplicationContext）创建之前**加载关键配置。

> ⚠️ 注意：**`bootstrap.yml` 仅在使用 Spring Cloud 时才有意义**。如果你的项目 **没有引入 Spring Cloud 相关依赖（如 Spring Cloud Config、Nacos、Consul 等）**，那么 `bootstrap.yml` **不会被加载**，也不会生效。

---

### 一、核心作用

`bootstrap.yml` 的主要用途是：

1. **配置外部配置中心的连接信息**
    - 例如：告诉应用去哪里拉取 `application.yml`（比如从 Git、Nacos、Consul 等）。
    - 因为这些配置本身是“获取其他配置的前提”，所以必须在主应用上下文初始化**之前**就准备好。

2. **设置应用名称、激活的 profile、加密/解密相关配置等**
    - 这些信息可能影响后续配置的加载逻辑。

3. **作为“引导上下文”（Bootstrap Context）的配置源**
    - Spring Cloud 会先创建一个 **Bootstrap ApplicationContext**，它比主应用上下文更早初始化。
    - Bootstrap Context 的配置来自 `bootstrap.yml`，并会被**合并到主上下文中**。

---

### 二、加载顺序（优先级）

Spring Boot + Spring Cloud 启动时的配置加载顺序如下（**越靠后优先级越高**）：

1. `bootstrap.yml`（来自 classpath）
2. `bootstrap-{profile}.yml`
3. `application.yml`
4. `application-{profile}.yml`
5. 命令行参数、环境变量等

> ✅ **关键点**：`bootstrap.yml` 中的配置会**早于** `application.yml` 加载，并且如果两者有相同属性，**`application.yml` 会覆盖 `bootstrap.yml`**（因为后加载）。但某些 Spring Cloud 组件（如 Config Client）**只读取 bootstrap 阶段的配置**，所以像 `spring.cloud.config.uri` 必须放在 `bootstrap.yml` 中。

---

### 三、典型使用场景示例

#### 场景：使用 Spring Cloud Config Server

```yaml
# bootstrap.yml
spring:
  application:
    name: user-service          # 应用名，用于从 Config Server 获取对应配置
  profiles:
    active: prod                # 指定环境
  cloud:
    config:
      uri: http://config-server:8888   # Config Server 地址
      enabled: true
```

- 应用启动时，先读取 `bootstrap.yml`；
- 根据其中的 `spring.cloud.config.uri` 和 `spring.application.name`，
- 去远程 Config Server 拉取 `user-service-prod.yml` 配置；
- 然后再初始化主应用上下文，使用拉取到的配置。

---

### 四、与 `application.yml` 的区别总结

| 特性                    | `bootstrap.yml`                                                              | `application.yml`                        |
|-----------------------|------------------------------------------------------------------------------|------------------------------------------|
| **加载时机**              | 应用上下文创建**之前**                                                                | 应用上下文创建**期间**                            |
| **用途**                | 配置“如何获取配置”（如配置中心地址）                                                          | 应用本身的业务配置                                |
| **是否需要 Spring Cloud** | 是（否则不生效）                                                                     | 否（Spring Boot 原生支持）                      |
| **优先级**               | 更早加载，但可被 `application.yml` 覆盖（部分组件除外）                                        | 后加载，覆盖 bootstrap 中同名属性                   |
| **常见内容**              | `spring.application.name`, `spring.cloud.config.*`, `spring.cloud.nacos.*` 等 | `server.port`, `datasource`, `logging` 等 |

### 五、注意事项

1. **Spring Boot 2.4+ 默认禁用了 bootstrap 上下文**
    - 如果你使用较新版本的 Spring Boot（≥2.4）和 Spring Cloud，需要**显式启用**：
      ```properties
      # 在 application.yml 或系统属性中添加
      spring.cloud.bootstrap.enabled=true
      ```
      或添加依赖：
      ```xml
      <dependency>
          <groupId>org.springframework.cloud</groupId>
          <artifactId>spring-cloud-starter-bootstrap</artifactId>
      </dependency>
      ```

2. **不要把普通配置放进去**
    - 只放**引导阶段必需的配置**，否则会增加启动复杂度。

3. **安全敏感信息**
    - 如配置中心的用户名/密码，也应通过环境变量或密钥管理服务注入，而非硬编码。

---

### 六、替代趋势（现代做法）

随着 Spring Cloud Alibaba、Nacos、Apollo 等注册配置中心的发展，以及 Spring Boot 2.4+ 对配置处理的改进，**很多场景已不再强制需要 `bootstrap.yml`**。例如：

- Nacos 客户端可以直接在 `application.yml` 中配置：
  ```yaml
  spring:
    application:
      name: user-service
    cloud:
      nacos:
        config:
          server-addr: 127.0.0.1:8848
  ```
  （前提是使用了支持自动装配的新版 starter）

但为了兼容性和明确职责，**在使用 Config Server 或需要严格分离“引导配置”时，仍推荐使用 `bootstrap.yml`**。

---

✅ **总结一句话**：
> `bootstrap.yml` 是 Spring Cloud 应用的“启动引导配置”，用于在主配置加载前，告诉应用**去哪里、用什么方式获取真正的配置**。
